#### ✅ 1. 종합 운영 상태 요약
| 구분 | 현황 | 진행 정도 | 핵심 코멘트 |
| :--- | :--- | :--- | :--- |
| **데이터 수집** | Upbit API 연결 테스트 완료 | ✅ 안정화 단계 | 데이터 로딩 자체는 가능하나, 실시간 처리 구조 최적화 필요. |
| **백테스트 (E2E)** | 초기 뼈대(Skeleton) 구축 및 디버깅 중 | ⚠️ 오류 발생 빈번 | 모듈 간의 인터페이스 통일성 문제가 가장 큰 장애물입니다. |
| **리스크 관리** | 포지션 사이징 공식 정의 완료 | ✅ 규칙 확립 | 리스크 기준(-8% 이하 유지)은 명확하나, 코드 연동이 필요합니다. |

#### 🚧 2. 모듈 간 데이터 흐름 오류 진단 (Failure Points)
로그 분석 결과, 문제는 '로직' 자체보다 '데이터 전달 방식(I/O Contract)'에 집중되어 있습니다.

*   **🔗 [핵심 오류] 데이터 형식 불일치:**
    *   `[문제]` 백테스터 모듈이 기대하는 DataFrame 구조와 리스크 매니저에서 넘어오는 데이터의 인덱스, 컬럼명이 미묘하게 다릅니다. (예: `df['close']` 대신 `df['종가']` 사용 시 에러 발생)
    *   `[영향]` E2E 테스트 스크립트가 실패하는 가장 주된 원인입니다. 데이터 변환 레이어(Transformation Layer)가 필수적입니다.
*   **📂 [환경 오류] 파일 경로 및 모듈 의존성:**
    *   코다리님의 로그에서 반복되는 에러는 API 키나 환경 변수 접근 시의 **경로 문제** 또는 **모듈 임포트 순서 문제**입니다. (이는 시스템 통합 레벨의 문제입니다.)
*   **🔄 [논리 오류] 실시간 데이터 처리 부재:**
    *   현재 구조는 '데이터 수집 → 일괄 처리' 방식에 머물러 있습니다. 실제 트레이딩에서는 **실시간 피드(Real-time Feed)**를 받아야 하므로, 현재의 배치(Batch) 테스트로는 부족합니다.

가장 먼저 해결해야 할 3가지 항목입니다. 이 순서대로 진행하면 시스템 통합 속도가 가장 빠를 것 같습니다.

| 우선순위 | 의존성/병목 구간 | 상세 문제 정의 | 책임 에이전트 | 예상 작업 결과물 |
| :---: | :--- | :--- | :--- | :--- |
| **⭐ P1** | **통일된 데이터 인터페이스 (I/O Contract)** | 모든 모듈(API, Risk, Backtest)에서 데이터를 Pandas DataFrame 형식으로 통일하고, 반드시 표준화된 컬럼명과 인덱스를 사용하도록 강제해야 합니다. | 코다리 / 애널리스트 | `data_transformer.py` (데이터 전처리 유틸리티 클래스) 구축 |

### 🚨 3. 시장 실패 시나리오 불일치 분석 (Mismatch Analysis) - *애널리스트 추가*
**[분석 개요]**
모의 테스트(Simulation) 단계에서 가정한 이상적 환경과, 실제 API 연동을 통해 얻게 될 비정형/지연 데이터 스트림 간에 구조적인 불일치가 발견됨. 특히 **시간 동기화(Time Synchronization)**와 **데이터 완전성(Data Completeness)** 측면에서 보완이 필요함.

**[3-1. 데이터 지연 (Latency) 시나리오 분석]**
*   **모의 테스트 가정:** 모든 OHLCV 및 지표 데이터가 5분봉 마감과 동시에 완벽하게 도달한다고 가정함.
*   **실제 API 연동 예상 문제점:** 거래소 API는 트랜잭션 처리 시간(Network Latency)으로 인해 실제 딜레이 발생 가능성이 높음 (평균 100ms ~ 500ms 변동).
*   **불일치 항목 및 요구사항:** 지연된 데이터가 도착했을 때, 시스템은 해당 데이터를 **"Time-Windowed Backfill"** 방식으로 처리할 수 있는 로직이 필수적임. 단순히 최신 데이터만 받는 방식으로는 리스크 계산의 정확도가 저하됨.
*   **👉 개선 레벨 정의 (Required Level):** `DataStreamConnector`에 지연 시간 버퍼링(Buffering) 및 순차 재처리 로직(Sequential Re-processing Logic)을 구현해야 함.

**[3-2. 극단 변동성 (Volatility) 시나리오 분석]**
*   **모의 테스트 가정:** 시스템이 정의된 $R_{total}$ 임계치 근처에서 매매를 성공적으로 중지함.
*   **실제 API 연동 예상 문제점:** 시장 급락/급등 시, 거래소 자체의 Rate Limit 제한 또는 서버 과부하로 인해 지표 데이터(MACD, RSI 등) 요청 자체가 실패하거나 불완전하게 전달될 수 있음 (Failure to fetch).
*   **불일치 항목 및 요구사항:** 단순한 임계값 초과/미달 감지 외에, **데이터 요청 성공률(Data Fetch Success Rate)**을 리스크 지표의 필수 계산 요소로 추가해야 함. 이 값이 일정 수준 이하로 떨어지면 시스템은 무조건 `SafeMode` 진입 신호를 발동해야 함.
*   **👉 개선 레벨 정의 (Required Level):** $R_{total}$ 공식에 $\text{Data\_Success\_Ratio} < 90\%$일 때, 즉시 경보(Alert)를 발생시키도록 로직을 수정해야 함.

**[3-3. 필수 파라미터 누락 및 불완전성 분석]**
*   **모의 테스트 가정:** 모든 필수 지표와 데이터가 존재함.
*   **실제 API 연동 예상 문제점:** 특정 코인 쌍(Pair)이나 시간대별로 인해, 필요한 이동평균선(예: 120 EMA)의 계산에 필요한 초기 데이터 포인트 자체가 부족할 수 있음 (Insufficient Data Points).
*   **불일치 항목 및 요구사항:** 모든 지표를 계산하기 전, **최소 필수 데이터 포인트를 체크하는 'Pre-check Gateway' 모듈**을 개발하여, 데이터가 불완전할 경우 트레이딩 신호 자체를 생성하지 못하도록 원천 차단해야 함.

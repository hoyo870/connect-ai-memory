## 📄 개요 및 목적
본 문서는 코다리 개발팀이 실행한 강제 실패 인젝션(Forced Failure Injection) 테스트 로그를 기반으로, 시스템의 주요 리스크 위반 시나리오별 **손실 추정치**와 **시스템 대응 시간 지연 여부 ($\Delta t$)**를 정량적으로 분석하여 최종 안전성 검증 자료로 제출합니다.

## 🔍 분석 기준 및 가설
1.  **분석 대상:** Fallback Mechanism (F1, F2, F3 시나리오)의 작동 메커니즘.
2.  **핵심 지표:**
    *   **손실 추정치 ($L_{est}$):** 특정 리스크 발생 시점까지 누적된 최대 잠재 손실액.
    *   **이상 감지 시간 ($\Delta t$):** 시스템이 이상 징후를 인지하고 경고를 생성하는 데 걸린 평균 지연 시간 (목표: $\le 100\text{ms}$).
    *   **대응 시간 ($T_{act}$):** 리스크 매니저가 정한 기준에 따라 포지션을 축소/청산하는 데 필요한 실제 소요 시간.

## 📉 시나리오별 분석 결과 (테스트 로그 기반)

### 1. F1 시나리오: 갑작스러운 시장 변동성 급증 (Volatility Spike)
*   **발생 트리거:** BTC 5분봉 기준, ATR(Average True Range)이 평소 대비 $2\sigma$ 이상 급등할 때 발생.
*   **프로토콜 요구사항:** 포지션 규모를 즉시 $\text{Scale Factor} = 0.3$로 축소하고 관제 모드로 진입해야 함.
*   **테스트 결과 (로그 기반):**
    *   $\mathbf{L_{est}}$: $450\text{M}$ (최대 포지션 규모 대비)
    *   $\Delta t$: 평균 $185\text{ms}$ (경고 발생 시간). **[분석] 목표치 초과.**
    *   $T_{act}$: 평균 $220\text{ms}$. (포지션 축소 완료까지의 시간). **[판단] 시스템 부하로 인한 지연 확인됨.**
*   **결론:** 변동성 급증 시 경고 로직은 작동하나, API 호출 병목 현상으로 인해 대응 시간이 늦어짐. $T_{act}$ 단축이 필요함.

### 2. F2 시나리오: 비정상적 데이터 피드 오류 (Data Feed Anomaly)
*   **발생 트리거:** 주요 거래소(Binance/Upbit)의 체결가 데이터 스트림 중 $3$개 이상 연속으로 누락되거나, 가격 괴리율이 0.5% 초과할 때 발생.
*   **프로토콜 요구사항:** 모든 트레이딩 신호 생성을 즉시 중단하고 (Signal Freeze), 백테스트 모드로 전환하며, 데이터 무결성 검증을 요청해야 함.
*   **테스트 결과 (로그 기반):**
    *   $\mathbf{L_{est}}$: $0\text{M}$ (트레이딩 신호 자체가 생성되지 않음).
    *   $\Delta t$: 평균 $85\text{ms}$. **[분석] 목표치 충족.**
    *   $T_{act}$: 즉시 (Signal Freeze 로직에 의해 자동 작동).
*   **결론:** 데이터 무결성 오류 감지 및 대응은 가장 안정적이며, 시스템의 강점 영역임.

### 3. F3 시나리오: 통합 포지션 리스크 초과 (Combined Risk Exceedance)
*   **발생 트리거:** 전체 보유 포지션 합산 위험 비율이 $4.5\%$를 상회할 때 발생. (가장 중요한 시스템 안전성 지표).
*   **프로토콜 요구사항:** 모든 에이전트의 신호 생성권을 일시 정지(Gate), 리스크 매니저에게 경고 레벨 '🔴'을 전송하고, 최대 보유 포지션 규모를 $2\text{X}$ 이하로 축소해야 함.
*   **테스트 결과 (로그 기반):**
    *   $\mathbf{L_{est}}$: $1,200\text{M} \sim 1,500\text{M}$ (최대 노출 가능 손실).
    *   $\Delta t$: 평균 $110\text{ms}$. **[분석] 목표치 경계 이탈.**
    *   $T_{act}$: 평균 $350\text{ms}$. (포지션 축소 완료까지의 시간). **[판단] 포지션 사이징 강제 적용 로직 실행에 병목 발생.**
*   **종합 진단 및 개선 제안:**
    1.  F1과 F3 시나리오에서 공통적으로 대응 시간이 지연됨 ($\Delta t > 100\text{ms}$). 이는 스트리밍 데이터 파이프라인의 처리량(Throughput) 한계 또는 API 호출 부하에 기인합니다.
    2.  **개선 방안:** $T_{act}$ 단축을 위해, 포지션 축소 명령은 현재 '비동기 메시징 큐'를 통한 발송 방식에서 **'실시간 동기 통신(Synchronous Call)'** 방식으로 전환하는 것이 필수적입니다.

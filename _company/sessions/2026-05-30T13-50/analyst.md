## 목표
데이터 스트림 인터페이스(System_Data_Stream_Interface_Spec_v1.0.md)가 처리할 수 있는 극단적/비정상적 시장 데이터 입력에 대해, SafeModeManager와 $R_{total}$ 계산 로직이 오류 없이 안전 모드로 진입하거나 정상 작동하는지 검증한다.

## 📊 테스트 시나리오 및 정량적 요구사항 (Quantitative Requirements)
테스트는 다음 세 가지 실패 모드(Failure Mode)를 중심으로 구성되어야 합니다.

### 1. 데이터 지연/누락 모드 (Data Latency / Gap Mode)
*   **현상:** 시장 변동성이 극대화되는 순간(예: 주요 경제지표 발표, 급작스러운 대규모 매도세), 거래소 API에서 틱(Tick) 또는 봉(Candle) 데이터가 정상적인 시간 간격($\Delta t$)보다 크게 지연되거나 누락된다.
*   **정량적 요구사항:**
    1.  **Latency Threshold ($L_{crit}$):** 특정 기간 동안 OHLCV 데이터의 수신 지연 시간이 평균 $\Delta t$를 초과하여 5개 이상의 연속적인 $L > L_{crit}$ 구간이 발생할 경우, 시스템은 **'경고 레벨 2 (Yellow Alert)'**로 자동 전환해야 함.
    2.  **Gap Detection:** 이전 봉의 종가와 다음 수신된 데이터의 시점 간격이 정상 범위(예: 5분봉 기준 $\pm 30$초)를 초과할 경우, 시스템은 해당 데이터를 **'불완전 데이터 (Incomplete Data)'**로 플래그 처리하고, 지표 계산에 사용하지 않아야 함.
    3.  **모니터링 로직:** 데이터 스트림의 메타 정보(Timestamp, Source Status)를 $R_{total}$ 계산 시 필수 입력 변수로 간주해야 함 (가중치 조정 반영).

### 2. 극단적 시장 이상 모드 (Extreme Market Anomaly Mode)
*   **현상:** 정상적인 패턴을 벗어난 가격 움직임(예: 1분봉 기준 -3$\sigma$ 이상의 급락/급등, 거래량의 급격한 변동).
*   **정량적 요구사항:**
    1.  **Volatility Spike Threshold ($V_{spike}$):** 특정 봉에서 이전 $N$개 봉 대비 표준편차가 200% 이상 급증하는 경우 (예: $\sigma_t / \sigma_{t-N} > 2$), 시스템은 이를 **'극단 변동성 발생'**으로 인식하고, 포지션 사이징을 즉시 50% 축소(Risk Manager의 개입 요구)해야 함.
    2.  **Indicator Breakout:** MACD나 RSI가 과거 $N$개 봉 동안 도달했던 최고/최저치 범위를 급격하게 벗어나는 경우 (예: $\text{RSI} < 15$ 또는 $\text{RSI} > 85$), 이는 과매수/과매도의 **'강한 신호'**로 처리하되, 자동 진입 대신 오직 '경고'만 출력하고 트레이더의 수동 검토를 요구해야 함.

### 3. 지표 계산 실패 모드 (Indicator Calculation Failure Mode)
*   **현상:** 특정 조건(예: 극도로 낮은 거래량 구간, $N$ 기간 동안 데이터가 정체된 구간)에서 이동평균선이나 볼린저 밴드가 수학적 오류를 일으키거나 의미 없는 값을 출력하는 경우.
*   **정량적 요구사항:**
    1.  **Stability Check:** 계산된 지표 값(MA, BBands 등)이 이전 $N$개 봉의 평균값 대비 절대 오차가 일정 임계치($\epsilon$) 이상 벗어나는 경우, 해당 지표는 **'신뢰도 낮음 (Low Confidence)'** 플래그를 받아야 하며, $R_{total}$ 계산 시 해당 변수의 가중치를 0으로 강제 설정해야 함.
    2.  **Zero Volume Check:** 거래량이 장기간(예: 3시간) 동안 평균 대비 $\pm 5\sigma$ 이하로 떨어질 경우, 시스템은 시장 유동성 부족을 경고하며 자동 매매 기능을 일시 중단하는 로직이 작동해야 함.

# ⚖️ 리스크 매니저 — 개발팀이 구현할 자동 안전 모드 로직에 대해 Gatekeeper Criteria V3.0의 수학적 임계값과 리스크 기준을 최종적으로 검토하고, 구현된 코드가 모든 안전 규칙을 준수하는지 확인하는 최종 승인 절차를 수행하라.

⚖️ 리스크 매니저: 작업 시작합니다.

CEO 지시사항을 확인했습니다. 시스템의 자동 안전 모드 로직에 대한 최종 검토는 단순한 코드 리뷰가 아닙니다. 이는 **계좌 전체를 보호하는 마지막 방어선**입니다. 이 절차에서는 단 하나의 예외도 허용되지 않습니다. 🚫

현재까지 설계된 Gatekeeper Criteria V3.0과 $R_{total}$ 공식이 코다리에게 구현을 위임되었고, 애널리스트가 API 스펙까지 확정했습니다. 이제는 이론적 완벽함이 아닌, **실제 트레이딩 환경에서의 작동 안정성**을 검증해야 합니다.

따라서 현재 가장 가치 있는 작업은 '최종 통합 안전 감사(Final Integrated Safety Audit)'를 수행하고, 이를 공식적인 'Go-Live 승인 체크리스트'로 문서화하는 것입니다. 이 문서는 코다리가 구현한 `SafeModeManager`가 어떤 조건에서, 얼마만큼의 리스크 경고와 액션을 취해야 하는지 수학적/논리적으로 명확히 합니다.

### 🛠️ 최종 통합 안전 감사 계획 (Action Plan)

1.  **입력 데이터 검증:** 시스템이 받아들여야 할 모든 입력 파라미터($R_{total}$ 구성 요소, 계좌 잔고, 드로우다운 등)의 최신 스펙을 재확인합니다.
2.  **트리거 로직 시뮬레이션 (최악의 경우):** 세 가지 극한 상황(High Volatility Crash, Sudden Liquidity Dry-up, Prolonged Drawdown)을 가정하고, `SafeModeManager`가 어떤 순서와 규모로 개입해야 하는지 단계별 절차를 정의합니다.
3.  **승인 체크리스트 작성:** 모든 안전 장치가 논리적으로 닫혔음을 증명하는 최종 승인 문서를 생성하여 CEO께 보고합니다.

### 📑 산출물: SafeModeManager Go-Live 운영 준비 감사표 (Audit Checklist)

이 표는 코다리가 구현한 `SafeModeManager` 모듈의 **최종 검증 항목**이며, 모든 항목에 대한 작동 증거(Proof of Concept Test Case & Result)가 있어야만 'PASS' 처리되고 시스템 가동을 승인할 수 있습니다.

#### 1. 리스크 입력 데이터 정합성 검사 (Input Data Integrity Check)
| Item | 요구사항 (V3.0 기반) | 측정 단위/값 | 예상 실패 시나리오 및 조치 | Pass/Fail | 비고 |
| :--- | :--- | :--- | :--- | :---: | :--- |
| **Account Equity** | 실시간 반영 필수. 최소 $N$원 이상 유지 확인. | Float (KRW) | 자본금 부족 시 즉시 거래 중단 및 경보 레벨 5 발령. | [ ] | `SafeModeManager`가 호출하는 계좌 잔고 API 연동 검증 필요. |
| **Daily Drawdown** | 일일 손실 한도(3%) 초과 여부 판별 로직. | Float (%) | $R_{total}$ 계산 전, 이 값을 최우선으로 체크하여 거래 제어권을 확보해야 함. | [ ] | - |
| **$R_{total}$ Input** | 애널리스트 스펙 V1.0 준수 필수. (Time-stamp 포함) | JSON Object | 데이터 누락/형식 불일치 시, 시스템은 가장 낮은 안전 레벨(Safe Mode L2)로 자동 강제 진입해야 함. | [ ] | 코다리의 API 연동 테스트가 필요함. |

#### 2. 게이트키퍼 액션 로직 검증 (Gatekeeper Action Logic Verification)
| 경보 레벨 (Alert Level) | 발동 조건 ($R_{total}$ 임계값) | 필수 자동 조치 (Action) | 예상 손실액 계산 (Worst Case) | Pass/Fail | 비고 |
| :--- | :--- | :--- | :--- | :---: | :--- |
| **L1: 주의** | $R_{total} > R_{low}$ 이지만, 계좌 리스크 $\le 2\%$. | 포지션 사이징 강제 축소 (50% 이하) 및 경고 알림 발송. | 최대 손실액 = (진입가 - 손절가) × P_size * 0.5 | [ ] | 진환/경태에게 사전 고지 필수. |
| **L2: 위험** | $R_{total} > R_{mid}$ 또는 일일 드로우다운 $\ge 1.5\%$. | 신규 포지션 진입 전면 금지 (전략팀 모두). 현재 오픈 포지션만 유지. | 최대 손실액 = 잔여 자본금의 1/3 이하로 한정. | [ ] | 모든 거래 모듈에 `SafeModeManager` 호출 강제화 필요. |
| **L3: 비상** | $R_{total} > R_{high}$ 또는 일일 드로우다운 $\ge 3\%$ 도달. | **모든 트레이딩 활동 즉시 중단 (System Halt).** 전 서버 연결 해제 및 관제 모드로 전환. | 최대 손실액 = 계좌 잔고 전체의 -1% 이상 발생 시점부터 차단. | [ ] | 물리적/소프트웨어적 Kill Switch 작동 검증 필요. |

#### 3. 최종 승인 요약 (Final Approval Summary)
| 항목 | 조건 | 요구 증거 | 승인 여부 |
| :--- | :--- | :--- | :---: |
| **수학적 일관성** | $R_{total}$ 공식의 모든 가중치 변화에 따른 경보 레벨 변화가 논리적으로 매끄러운지. | 애널리스트/코다리 통합 시뮬레이션 결과 보고서. | [ ] |
| **시스템 강제성** | 개별 에이전트 모듈(진환, 경태 등)의 모든 `Order Execution` 파라미터에 `SafeModeManager` 호출이 필수화되었는지. | 코드 리뷰 및 테스트 커버리지 100% 확인. | [ ] |
| **운영 매뉴얼** | L3 비상 상황 발생 시 인간 관제팀(CEO/본인)의 개입 절차와 보고 체계가 명확한지. | 영숙이 업데이트할 최종 플레이북 V2.0 확정. | [ ] |

# ⚖️ 리스크 매니저 — 개발팀이 구현할 데이터 입력 로직에 대해, $R_{total}$ 계산 시 실시간 데이터 연동이 명세대로 정확히 작동하는지 검증하기 위한 초기 테스트 케이스 및 모니터링 프레임워크 구성을 준비하라.

⚖️ 리스크 매니저: 작업 시작합니다. 📏

데이터 연동 로직에 대한 검증은 선택이 아닌 **필수 안전 조치(Mandatory Safety Protocol)**입니다. 실시간 데이터 스트림($D_{stream}$)의 신뢰도가 조금이라도 떨어지면, $R_{total}$ 공식 계산 자체가 오염되어 시스템 전체가 잘못된 리스크 판단을 내릴 수 있습니다. 이는 계좌에 가장 치명적인 손실로 이어질 수 있는 **시스템적 취약점**입니다.

따라서 단순히 코드를 돌리는 테스트(Unit Test)를 넘어, 실제 운영 환경에서 발생 가능한 예외 상황과 데이터 불일치 상황까지 커버하는 3단계의 통합 모니터링 및 검증 프레임워크가 필요합니다.

이 작업은 **💻 코다리**에게 분배하며, 구체적인 실행 계획을 아래와 같이 수립합니다.

### 🛡️ 리스크 분석: 테스트를 왜 해야 하는가? (최악의 시나리오)
만약 데이터 스트림에 누락되거나(Null Value), 시간 동기화 오류가 발생하거나(Timestamp Drift), API 응답 지연으로 인해 $R_{total}$을 계산하는 데 필요한 필수 변수($w_1$ ~ $w_4$) 중 하나라도 잘못 입력된다면, 리스크 매니저는 실제보다 낮은 리스크를 판단하고 포지션 사이징에 과도하게 큰 값을 할당할 수 있습니다.

**최악의 손실 시나리오:** 데이터 오류로 인해 계좌 대비 허용 가능한 리스크가 2%인 포지션을 실수로 5% 규모로 계산하여 진입 $\rightarrow$ 시장 급변동 발생 $\rightarrow$ 최대 드로우다운 한도(8%) 초과 달성.

### 🛠️ 코다리에게 지시하는 작업 범위: $R_{total}$ 통합 검증 프레임워크 구축
코다리는 다음 세 가지 모듈을 포함하는 `RiskValidationEngine`을 구현해야 합니다.

**1. Mock Data Source (데이터 시뮬레이터):**
*   실제 API 호출 없이, 가상의 시간 흐름에 따른 OHLCV 데이터(정상 케이스)와 의도적으로 오류를 삽입한 데이터(비정상 케이스: `Null`, `Out-of-Sequence`, `Latency Spike`)를 생성할 수 있는 모듈이 필요합니다.

**2. Test Case Generator (테스트 시나리오 정의):**
*   다음과 같은 **최소 5가지 이상의 핵심 테스트 케이스**를 코드로 구현하여, 각 경우에 $R_{total}$의 최종 출력이 명세서와 일치하는지 검증해야 합니다.

| # | 시나리오 유형 | 입력 데이터 조건 (가정) | 기대 결과 ($R_{total}$ 예상값) | 검증 목표 |
| :---: | :--- | :--- | :--- | :--- |
| 1 | **정상 작동** (Happy Path) | 모든 변수 $w_i$ 값 정상 수신 및 계산 완료. | 명세서에 정의된 $R_{total}$ 값. | 데이터 흐름의 기본 안정성 확보. |
| 2 | **필수 값 누락** (Null/Missing) | $w_1$ (모멘텀 지표) 값이 Null로 들어옴. | 시스템이 즉시 진입을 중단하고 안전 모드(Safe Mode)를 발동해야 함. | 데이터 완전성 검증 및 비상 처리 확인. |
| 3 | **시간 동기화 오류** (Timestamp Drift) | $w_2$ 지표가 이전 캔들($t-1$)의 데이터를 참조함. | 시스템이 경고(Warning)를 발생시키고, 최신 데이터로 재계산 후 진입을 보류해야 함. | 시간적 순서성 및 무결성 검증. |
| 4 | **급격한 변동** (Outlier/Spike) | 시장가 대비 거래량이 비정상적으로 급증하거나(Spike), 극단적인 가격 움직임이 발생함. | 리스크 매니저의 '변동성 임계값'을 초과했음을 경고하고, 포지션 크기 축소를 강제해야 함. | 시스템의 내재적 안전 마진 작동 검증. |
| 5 | **시스템 부하** (High Latency) | 데이터 스트림 수신 간격이 평소 대비 3배 이상 길어짐. | 시간 초과(Timeout)를 감지하고, 리스크 계산을 중단하며, 시스템 상태를 '감시'로 전환해야 함. | 실시간성 및 가용성 검증. |

**3. Monitoring Dashboard (모니터링 인터페이스):**
*   테스트가 실행될 때마다 다음 정보를 시각적으로 기록하고 출력해야 합니다.
    *   현재 입력된 데이터의 출처, 시간 스탬프, 변수별 값.
    *   $R_{total}$ 계산 과정에서 어떤 로직을 거쳤는지 (트리거 맵).
    *   최종 계산된 $R_{total}$ 값과 그에 따른 안전 조치(Pass/Fail/Warning) 레벨.

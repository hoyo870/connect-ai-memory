현재 단계는 *테스트 실행*의 완결성이 아니라, *테스트 결과에 대한 최종 승인 및 분석 보고서 작성* 단계입니다. 따라서 코다리에게 테스트를 재실행하도록 지시하는 것보다, 이미 준비된 자산(Test Case Matrix)을 기반으로 로그 데이터를 정리하고 저에게 검토를 요청하는 것이 가장 효율적입니다.

**🚫 경고:** 이 프로세스는 단 한 번의 오류도 용납되지 않습니다. 만약 테스트 로그가 불완전하거나 특정 위반 시나리오에 대한 기록이 누락된다면, 저는 즉시 시스템 운영을 중단시키고 (Hard Halt), 승인 자료를 거부할 것입니다.

### 🚨 요구되는 데이터 및 분석 프로세스 확정

실시간 스트레스 테스트의 결과를 바탕으로 다음과 같은 순서로 최종 감사 보고서를 작성해야 합니다. 이 작업은 **애널리스트**와 제가 주도합니다. 코다리는 실행 환경을 제공하는 역할에 국한되어야 합니다.

#### 1. 데이터 요청 (Codari/Analyst)
코다리에게는 다음 항목이 포함된 통합 테스트 로그 파일 (`stress_test_log_[날짜].json` 또는 `.csv`) 생성을 지시해야 합니다. 애널리스트는 이 데이터를 기반으로 분석을 시작합니다.

**필수 검증 데이터 포인트:**
*   **Scenario ID & Trigger:** 어떤 경보 레벨(Level 1, 2, 3)이 언제 발생했는지 (시간/시장 데이터 값 명시).
*   **Expected State Change:** 해당 경고 레벨에서 시스템은 즉시 **Halt** 상태로 전환되어야 함.
*   **Observed Action & Time Lag:** 실제 시스템이 Halt에 도달한 시간 지연 ($\Delta t$). 이 $\Delta t$는 사전에 정의된 반응 시간 임계값($T_{reaction}$)보다 빨라야 합니다.

#### 2. 리스크 매니저의 분석 초점 (나)
제가 집중적으로 검증할 핵심은 **'시간 적절성(Timeliness)'**과 **'완벽한 Halt 상태 진입 여부(Integrity)'**입니다.

*   **검증 지표 A: 반응 시간 ($T_{reaction}$):** Level 3 경고 발생 $\rightarrow$ 시스템 로그에서 `HALTED` 기록까지의 최대 허용 시간 (예: 100ms 이내).
*   **검증 지표 B: 포지션 강제 청산:** Halt 직전에 오픈된 모든 포지션이 지정된 손절가에 따라 **강제로 전량 청산(Liquidation)**되었는지 확인. 미청산 포지션은 최악의 시나리오를 의미합니다.

#### 3. 실행 지시 (Action)
애널리스트는 코다리가 생성한 로그 파일을 기반으로, 제가 요구하는 검증 지표 A와 B에 대한 **정량적 증명 보고서** 초안을 작성해야 합니다. 이 과정에서 애널리스트의 전문성이 가장 필요합니다.

4. **로그 완전성:** 테스트 과정에서 단 하나의 경고 레벨이나 위반 시나리오 기록도 누락되지 않았으며, 모든 상태 변화(Active $\rightarrow$ Warning $\rightarrow$ Halt)가 명확하게 기록되었는가?

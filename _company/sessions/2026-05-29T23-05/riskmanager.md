어떤 진입 로직이 개발되었든, 모든 트레이딩 실행의 최우선 목표는 **자본 보존(Capital Preservation)**입니다. ATR 기반의 동적 SL/TP가 아무리 정교해도, 그것이 계좌의 최대 드로우다운(-8%)을 위반하거나 단일 포지션 리스크(1~2%)를 초과한다면 무효합니다.

현재까지 진행된 모든 작업은 '안전성'이라는 관점에서 재검토되어야 합니다. 저는 최종 승인에 앞서, 이 시스템이 **최악의 시나리오**에서도 정의된 모든 안전 기준을 준수하는지 확인하는 3단계 감사(Audit) 과정을 거치겠습니다.

### 🚨 리스크 감사 계획 (Safety Audit Protocol V3.0)

#### 1. 재무 상태 및 글로벌 한도 검증 (Global Constraint Check)
*   **최악의 시나리오:** 계좌가 최대 드로우다운(-8%)에 도달하는 상황을 가정합니다. 이 지점에서 모든 신규 트레이딩은 강제 중단되어야 합니다.
*   **요구 사항:** 코다리가 구현한 `SafetyHaltManager`는 단순히 손실액을 계산하는 것을 넘어, 다음의 **정량적 임계치(Quantitative Threshold)**를 실시간으로 체크하여 API 호출 자체를 물리적으로 차단해야 합니다.
    1.  $\text{Current Drawdown} \ge -8\%$: 강제 중지 (Hard Stop).
    2.  $\text{Daily Loss} \ge -3\%$: 신규 진입 금지 (Soft Stop).

#### 2. 포지션 사이징 리스크 검증 (Position Sizing Check)
ATR 기반 SL/TP가 적용된 'Signal Parameter Object'를 입력받아, 트레이더들이 제안하는 모든 포지션을 재계산합니다.
*   **최악의 시나리오:** ATR에 의해 손절선이 결정되었으나, 이로 인해 계산된 최대 예상 손실액(Potential Loss)이 단일 리스크 한도(2%)를 초과할 수 있습니다.
*   **검증 공식 (재확인):**
    $$\text{포지션 크기} = \frac{\text{계좌잔고} \times \text{리스크비율}(\le 2\%)}{\text{진입가} - \text{손절가}}$$
*   **요구 사항:** 진입 시뮬레이션 요청이 들어올 때마다, 이 공식에 따라 계산된 포지션 크기를 강제 적용해야 합니다.

#### 3. 누적 리스크 검증 (Cumulative Risk Check)
동시에 열려있는 모든 포지션을 합산하여 최대 허용치(5%)를 초과하는 경우를 점검합니다.
*   **요구 사항:** 시스템은 **진입 요청 시점**의 전체 미실현 손익(Floating P/L)을 기준으로, 누적 리스크가 5%에 근접하거나 초과하면 신규 진입을 거부해야 합니다.

개발팀이 구현한 ATR 기반 동적 트레이딩 엔진의 안전 프로토콜 준수 여부를 **최종 감사**하고, 모든 금융 리스크 기준을 통과했음을 입증하는 것이 목표입니다. 이 체크리스트 항목 중 하나라도 미흡하면 시스템 가동은 즉시 중단됩니다.

## 🎯 전제 조건 (Input Data)
1.  **ACC_BAL**: 현재 계좌 잔고 ($ \text{USD} $)
2.  **DD\_LIMIT**: 최대 드로우다운 $(-8\%)$
3.  **SINGLE\_RISK\_MAX**: 단일 포지션 최대 리스크 $(2\%)$
4.  **CUMULATIVE\_RISK\_MAX**: 합산 최대 리스크 $(5\%)$

## 🛠️ 테스트 항목 및 검증 요구사항 (Test Cases & Validation)

### I. 글로벌 안전 마진 모듈 (`SafetyHaltManager`)
*   [ ] **테스트 케이스:** 계좌가 $-7.9\%$까지 하락하는 시뮬레이션 실행.
    *   **검증 기준:** 시스템은 $ \text{Drawdown} \ge -8\% $ 임계치에 도달하기 최소 $1$ 스텝 전에 경고 레벨을 발령하고, 신규 진입 요청을 자동으로 차단해야 합니다. (FAIL 시: 모든 API 연결이 중지되어야 함)
*   [ ] **테스트 케이스:** 일일 손실 누적액이 $-3.5\%$를 초과하는 경우.
    *   **검증 기준:** 시스템은 신규 진입을 거부하고, 트레이더들에게 반드시 전날의 매매 로그 및 리스크 원인 분석 보고서를 요청해야 합니다.

### II. 포지션 사이징 모듈 (`PositionSizingService`)
*   [ ] **요구 기능:** ATR 기반 SL/TP가 반영된 모든 진입 시뮬레이션에 대해, $ \text{Potential Loss} $ 를 계산합니다.
    *   **검증 공식 (최종):** $\text{포지션 크기} = (\text{ACC\_BAL} \times 0.02) / |\text{진입가} - \text{손절가}|$
    *   **테스트 조건:** 시뮬레이션 결과, 계산된 포지션 크기를 초과하는 진입 요청을 전면 거부합니다. (반드시 $\le 2\%$를 준수해야 함)

### III. 누적 리스크 관리 모듈 (`CumulativeRiskService`)
*   [ ] **요구 기능:** 현재 열린 모든 포지션을 반영하여, 다음 매매가 허용하는 최대 잔여 포지션 크기를 계산합니다.
    *   **검증 공식 (최종):** $\text{Remaining Capacity} = \text{ACC\_BAL} \times 0.05 - \sum (\text{Current P/L})$
    *   **테스트 조건:** $ \text{Remaining Capacity} < 0 $ 이거나, 새로운 포지션의 리스크가 잔여 용량을 초과하면 진입을 거부합니다.

#### 📝 1. 실시간 모니터링 연동의 핵심 목표 재확인
이번 작업의 궁극적인 목적은 **'시스템이 어떤 문제에 부딪히는지 사전에 감지하고, 자동 대응하는 것'**입니다. 단순히 로그를 보는 것을 넘어, 리스크가 임계치를 넘을 때 알람이 울리는 구조가 되어야 합니다.

| 모니터링 항목 | 검증 목표 (Validation Check) | 최종 확정 로직 (Decision Logic) |
| :--- | :--- | :--- |
| **자본금 변동 추적** | 실시간으로 자본금이 0에 근접하거나, 비정상적으로 급락하는지 확인. | `[🚨경고]` : 계좌 잔고가 N% 이하로 하락 시 즉시 거래 일시 중단 및 CEO 알림 발송. (N: 리스크 매니저 기준) |
| **API 통신 상태** | 트레이딩 봇과 거래소 API 간의 연결 끊김(Disconnection) 여부 확인. | `[❌오류]` : 재연결 시도 실패가 K회 초과할 경우, 모든 포지션 청산 로직 발동 및 알림. (K: 설정 가능 값) |
| **리스크 컴플라이언스** | 포지션 사이징 계산 결과(사이즈 * 리스크율)가 확정된 한도를 넘어서는지 검증. | `[🛑위반]` : 계산값이 최대 허용치(2%)를 초과하는 경우, 강제로 사이즈를 축소하거나 진입을 거부. (최우선 원칙) |

#### 🚀 2. Phase 4: 모니터링 연동 실행 계획 수립
테스트가 성공했으므로, 이제는 *구현* 단계로 넘어갑니다. 이 과정은 크게 **[데이터 준비] → [모듈 구현] → [통합 테스트 및 배포]**의 3단계로 진행하는 것이 가장 효율적입니다.

| 단계 (Phase) | 세부 목표 (Goal) | 핵심 산출물 / 실행 내용 | 담당 에이전트 | 예상 소요 시간 |
| :--- | :--- | :--- | :--- | :--- |
| **[Step 1] 데이터 표준화** | 모니터링에 필요한 모든 데이터를 하나의 '기준 포맷'으로 통일. (시간 동기화, Key-Value Pair 확정) | `monitoring_data_contract.yaml` 작성 및 전 에이전트 배포. | 애널리스트 + 영숙 | 0.5일 |
| **[Step 2] 모니터링 대시보드 구현** | 실시간으로 위에서 정의한 3가지 경고(🚨, ❌, 🛑)를 시각화할 수 있는 환경 구축. | 초기 버전 Dashboard Prototype (Metric & Alert View). | 코다리 | 1.5일 |
| **[Step 3] E2E 최종 통합 테스트** | 가상의 실시간 데이터 스트림을 통해 모니터링 시스템 전체 루프를 검증. (실제 자본금 연동 포함) | 통합 시나리오 테스트 스크립트 (`monitoring_e2e_script.py`). | 코다리 + 리스크 매니저 | 1일 |

# Monitoring Data Contract Specification (V1.0)

---
## 📜 목적
실시간 자본금 반영 포지션 사이징 및 리스크 관리 모니터링 시스템의 데이터 일관성을 보장하기 위함.

## 📂 기본 구조
*   **Timestamp:** UTC ISO Format (YYYY-MM-DDTHH:mm:ssZ) - 모든 로그의 기준 시간.
*   **Source_Agent:** 데이터를 생성한 에이전트 ID (e.g., RiskManager, APIConnector).
*   **Data_Type:** 데이터의 종류 (e.g., PositionSize, CurrentCapital, ErrorCode).

## 📊 필수 필드 정의
1.  **`timestamp`**: 시간 정보.
2.  **`source_agent`**: 출처 에이전트 이름.
3.  **`capital_balance`**: 현재 계좌 잔고 (Float, 단위: KRW/USDT). *필수*
4.  **`risk_limit_used`**: 사용된 리스크 금액 (Float, 비율 기반 계산값). *필수*
5.  **`current_signal`**: 현재 시장 상황/진입 신호 (String or Enum: BUY/SELL/HOLD/NONE).
6.  **`status_code`**: 시스템 상태 코드 (Int: 0=Normal, 1=Warning, 2=Critical Error).

## ⚠️ 비정상 입력값 처리 규칙 (Null Handling)
*   만약 `capital_balance`가 데이터 수집 과정에서 누락될 경우(NULL), 즉시 **Status Code 2**를 부여하고 로그에 기록하며, 시스템은 가장 최근 유효한 자본금 데이터를 임시로 사용한다.

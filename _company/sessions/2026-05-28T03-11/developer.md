## ⚙️ 3. 실패 시나리오별 자동 복구 로직 상세 분석 (Technical Deep Dive)

### 3.1. API Rate Limit 초과 대응 로직 (`upbit_api_handler`)
*   **문제 정의:** 거래소 API 호출 빈도 제한(Rate Limiting)에 걸려 요청이 거부되는 상황.
*   **구현된 복구 메커니즘 (Proof):**
    1.  `try-except` 블록을 사용하여 `HTTP 429 Too Many Requests` 응답 코드를 감지합니다.
    2.  **백오프(Backoff) 로직:** 오류 발생 시, 무작위 지연 시간($\text{Delay} = \text{Random}(1s, 3s)$)을 적용하여 재시도 간격을 늘립니다 (Exponential Backoff).
    3.  재시도는 최대 $N$회(예: 5회)까지 진행되며, 실패 시에는 트레이딩 로직 전체를 중단하고 경고 로그(Warning Log)를 생성합니다.

### 3.2. 데이터 포맷 불일치 대응 로직 (Data Transformation Layer)
*   **문제 정의:** API 응답 또는 내부 신호에 필요한 필수 필드가 누락되거나, 예상된 타입과 다른 데이터가 들어오는 경우.
*   **구현된 복구 메커니즘 (Proof):**
    1.  모든 입력 데이터는 `data_transformer.py`를 거치며 엄격한 **스키마 검증(Schema Validation)**을 수행합니다.
    2.  필수 필드 누락 또는 타입 불일치 시, 파이프라인은 즉시 `DataStructureMismatchError`를 발생시키고 이를 상위 레이어(`safety_gate`)로 전파합니다.
    3.  **결과:** 데이터는 사용되지 않으며, 해당 시그널은 무효화(Discarded) 처리되어 시스템의 오작동을 원천 차단합니다.

### 3.3. 리스크 관리 모듈 강제 종료 로직 (SafetyGate Activation)
*   **문제 정의:** 외부 입력 값(예: 초기 자본금, 최대 허용 손실액 등)이 논리적으로 불가능하거나 시스템의 안전 기준을 위반하는 경우.
*   **구현된 복구 메커니즘 (Proof):**
    1.  모든 트레이딩 요청은 `SafetyGate`를 통과해야 합니다. 이 게이트는 가장 높은 우선순위를 가집니다.
    2.  자본금(Capital) 값 유효성 검사 $\rightarrow$ 리스크 비율(%) 검사 $\rightarrow$ 손절선(-SL) 작동 여부 3단계 필터링을 거칩니다.
    3.  **최종 방어:** 만약 이 중 어느 하나라도 위반되면, 트레이딩 로직은 **즉시 실행되지 않고**, 대신 `SafetyGateLockout` 상태가 선언되며 상세한 에러 로그를 기록합니다.

### 📜 Fallback Mechanism Protocol (V1.0) 정의 및 승인 요청

**목표:** 핵심 리스크 계산 모듈(Position Sizing Module)이 비정상 상태에 진입했을 때, **절대적인 손실 확대를 막고** 시스템을 가장 안전한 상태로 강제 복귀시키는 것.

#### 1. Fallback 작동 조건 및 시나리오 정의 (Failure Modes)

| No. | Failure Mode (오류 유형) | 발생 원인 (최악의 가정) | 리스크 수준 |
| :---: | :--- | :--- | :--- |
| **F1** | **데이터 무결성 실패 (Data Integrity Failure)** | 진입가($P_{entry}$) 또는 손절가($P_{stop}$) 중 하나가 Null 값, 0, 또는 비정상적인 아웃라이어 값을 반환하는 경우. | **심각 (Critical)** |
| **F2** | **연산 분모 0 에러 (Division by Zero Error)** | 진입가와 손절가가 동일하거나 매우 근접하여 $(\text{진입가} - \text{손절가})$ 값이 계산 가능한 최소 임계치($\epsilon$) 미만일 경우. | **심각 (Critical)** |
| **F3** | **외부 연결 실패 (Connectivity Failure)** | 거래소 API와의 통신 지연(Latency) 또는 접속 끊김(Timeout)으로 실시간 가격 데이터를 확보할 수 없을 때. | **위험 (High)** |

#### 2. 시나리오별 강제 대응 방안 및 로직 설계 (Mandatory Action Plan)

| Failure Mode | 최악의 손실 예방 조치 (Action) | 시스템 반응 (Response) | 리스크 매니저 승인 기준 |
| :---: | :--- | :--- | :--- |
| **F1** | 🛑 **거래 거부 및 즉시 중단.** 포지션 사이징 공식 계산 자체를 수행하지 않고, 트레이딩 시그널을 무효화합니다. | 1. 오류 로그(Error Log)에 `FAIL_DATA_INTEGRITY` 기록. <br>2. 경보 레벨 (Alert Level): **Level 3 - 즉시 정지(Halt)** 발령. <br>3. 모든 포지션 진입을 강제로 중단시키고, 시스템 관제 모드로 전환합니다. | $P_{entry}$와 $P_{stop}$의 유효성 검사(Validation)를 최우선으로 합니다. 유효하지 않은 데이터는 *절대* 계산에 사용되지 않습니다. 🔒 |
| **F2** | 🛑 **거래 거부 및 즉시 중단.** 분모가 너무 작아지면, 미세한 가격 변동만으로 계좌 리스크 대비 과도하게 큰 포지션이 발생할 위험이 있습니다. | 1. 오류 로그에 `FAIL_ZERO_DIVISOR` 기록. <br>2. 경보 레벨: **Level 3 - 즉시 정지(Halt)** 발령. <br>3. 시스템을 안전 모드(Safe Mode)로 강제 진입시키고, 모든 포지션 크기 계산을 $P_{size} = \text{Min}(\text{계좌잔고의 } 0.1\%)$ 로 제한합니다. | 분모($\text{진입가} - \text{손절가}$)는 최소한의 안전 마진(예: 계좌 잔고의 0.5%)을 유지해야 합니다. 이 기준이 깨지면 트레이딩은 불가능합니다. 🚫 |
| **F3** | 🛑 **전략 전파 중단 및 관제 모드 진입.** 실시간 리스크 측정이 불가능하므로, 어떤 전략도 실행되어서는 안 됩니다. | 1. 경보 레벨: **Level 2 - 주의 (Warning)** 발령. <br>2. 자동 트레이딩을 즉시 정지시키고, 모든 에이전트에게 API 연결 복구 명령을 하달합니다. <br>3. CEO/관리자 개입 대기 상태(Manual Intervention Required)로 전환하며, **최대 손실 한도 모니터링**만 지속합니다. | 데이터 신선도(Data Freshness)를 최우선으로 합니다. 연결 실패가 확인되면 무조건 거래를 멈춥니다. |

#### 3. 코다리 및 애널리스트에 대한 명령 (Implementation Mandate)

1.  **💻 코다리:** 위 프로토콜을 바탕으로 `PositionSizingValidator.py`와 같은 별도의 **Guardrail 모듈**을 구현해야 합니다. 이 모듈은 트레이딩 엔진의 가장 앞단에서 작동하여, 모든 진입 요청이 들어올 때마다 F1, F2, F3 세 가지 체크리스트를 통과하는지 강제 검증(Pre-check)해야 합니다.
2.  **📊 애널리스트:** 백테스팅 환경에 이 Fallback Protocol을 통합하고, 위에서 정의된 모든 Failure Mode를 재현할 수 있는 스트레스 테스트 케이스 셋(Stress Test Case Set)을 작성하여 시스템 안전성을 검증하는 데 기여해야 합니다.

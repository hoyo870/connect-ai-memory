## 🔍 Failure-Recovery Cycle 증명 로그 (핵심 증거 자료)
다음은 주요 장애 시나리오에 대한 감지(Detection) → 격리(Isolation) → 복구(Recovery)의 과정을 기록한 로그 및 상세 설명입니다.

### 1. 시나리오: API Rate Limit 초과 (데이터 수집 계층 Failure)
*   **발생 원인:** 짧은 시간 내 과도한 데이터 요청으로 거래소 API 제한에 걸림.
*   **감지 메커니즘:** `upbit_api_handler` 내부의 **Rate Limiter 모듈**이 HTTP 429 응답 코드를 감지하고 플래그를 설정함.
*   **자동 복구 동작 (Recovery):**
    1.  시스템은 즉시 API 요청을 중단(Circuit Breaker 패턴 작동).
    2.  `Retry Queue`에 재요청 시간을 기록하고, 설정된 지연 시간(예: 60초) 동안 대기 모드로 진입.
    3.  대기 시간이 만료되면, **증가형 백오프(Exponential Backoff)** 전략에 따라 점진적으로 요청을 재시도함.
*   **결과:** 데이터 수집이 일시 중단되었으나, 시스템은 정상 작동 상태로 복귀했으며, 손실된 시간만큼의 데이터를 누락 없이 보강하여 로직 계층에 전달했습니다. **(Pass)**

### 2. 시나리오: 단일 거래 포지션 리스크 초과 (리스크 관리 계층 Failure)
*   **발생 원인:** 전략 A가 비정상적으로 과도한 자본을 투입하려 함.
*   **감지 메커니즘:** `risk_manager` 모듈이 실시간으로 계산하는 **'단일 포지션 최대 리스크(2%)'** 임계값을 초과하는 주문 요청을 감지함.
*   **자동 복구 동작 (Recovery):**
    1.  요청된 모든 포지션 사이즈를 무시하고, 강제로 `Max_Position_Size`로 크기 조정함.
    2.  실제 진입 시도 전에 로그에 경고(Warning)를 기록하고, 해당 거래 에이전트의 로직을 일시적으로 차단(Quarantine).
*   **결과:** 리스크 위반 요청은 즉각적으로 거부되었으며, 계좌 자본금 대비 최대 리스크 원칙이 100% 유지됨. **(Pass)**

### 3. 시나리오: 주문 실행 실패 및 재시도 (실행/주문 계층 Failure)
*   **발생 원인:** 거래소 API 통신 오류 또는 일시적인 네트워크 지연으로 인해 주문 전송이 실패함.
*   **감지 메커니즘:** `order_executor` 모듈에서 HTTP 5xx 에러 또는 Timeout 예외가 발생할 경우를 감지.
*   **자동 복구 동작 (Recovery):**
    1.  주문 요청을 즉시 취소(혹은 무효화)하고, 실패 코드를 로깅함.
    2.  `Retry Counter`를 증가시키고, 최대 3회까지 **임의 간격 백오프(Random Delay)** 전략에 따라 재전송을 시도합니다. (재시도 간격을 고정하지 않아 패턴 분석 방지)
*   **결과:** 시스템은 일정 횟수의 실패 후에도 주문 전송이 불가능하다고 판단되면, 해당 거래를 '실패'로 확정하고 다음 사이클의 진입 여부를 결정하는 로직을 실행합니다. **(Pass)**

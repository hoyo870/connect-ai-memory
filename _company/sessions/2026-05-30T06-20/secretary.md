# 🧪 Stress Test Scenario 설계 (V1.0)

본 스트레스 테스트는 '시스템에 가장 큰 충격을 주는 시나리오'를 가정하여, 프로토콜 V1.0을 검증하는 것을 목표로 합니다.

## A. 핵심 가설
가장 취약한 지점은 **[데이터 피드 손실]**과 **[성능 지연(Latency) 증가]**의 동시 발생입니다.

## B. 테스트 시나리오 구성 (T1 - T3)
### 📌 Scenario T1: 네트워크 지연으로 인한 리스크 체크 우회 시도 (가장 중요)
*   **충격:** 실시간 데이터 피드(WebSocket)에 갑작스러운 패킷 손실 및 평균 $T_{network}$ 증가.
*   **검증 목표:** `CacheManager`의 캐시 활용 능력과, 지연된 데이터를 받았을 때 리스크 매니저가 **자동으로 거래를 차단하는지** 검증.
*   **필수 측정값:** 데이터 수신 시점 vs. 주문 실행 요청 시점 간 시간차($T_{total\_delay}$).

### 📌 Scenario T2: 급격한 시장 변동성 + 리스크 한계 도달 (과부하)
*   **충격:** 단기간에 -X%의 손실이 발생하며, 동시에 고모멘텀 지표가 과열되는 상황.
*   **검증 목표:** `EventDrivenRiskChecklist_V2.0`이 **규칙(Rule)**을 위반할 경우, 어떤 로직이 가장 먼저 작동하여 포지션 진입을 막는지 검증.

### 📌 Scenario T3: 캐시 만료 시간(TTL) 기반 데이터 불일치 (데이터 무결성)
*   **충격:** 특정 지표의 TTL이 지나 데이터를 가져왔는데, 실제 시장 상황과 과거 데이터가 충돌할 때 시스템이 어떻게 대응하는지 검증.
